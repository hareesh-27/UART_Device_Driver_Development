## UART RX Interrupt - Basic Concept

In the previous UART RX Polling project, whenever we wanted to receive data, we continuously checked the RXNE flag using a while loop.

```c
while(!(USART2->SR & SR_RXNE)){}
```

This technique is called polling.

The CPU continuously checks the RXNE flag until data arrives.

The flow was:

```text
CPU checks RXNE
↓
RXNE = 0
↓
Check again
↓
RXNE = 0
↓
Check again
↓
RXNE = 1
↓
Read data
```

The disadvantage of polling is that the CPU keeps waiting and continuously checking the flag. During this time, the CPU cannot perform other useful tasks.

To solve this problem, we use interrupts.

Instead of continuously checking the RXNE flag, we tell the hardware:

```text
When data arrives,
don't make me keep checking.
Simply interrupt me.
```

The flow becomes:

```text
CPU doing other work
↓
Data arrives
↓
RXNE becomes 1
↓
USART generates interrupt
↓
CPU pauses current task
↓
Interrupt Service Routine (ISR) executes
↓
Data is processed
↓
CPU returns to previous task
```

Therefore, in interrupt mode, the CPU does not waste time continuously checking a flag.

## uart_rx_interrupt_init()

Coming to uart_rx_interrupt_init(), we have to add 2 extra pieces.

### i.

One is to enable RXNE interrupt by setting the 5th bit (RXNEIE bit).

```c
USART2->CR1 |= CR1_RXNEIE;
```

If we set this bit, whenever the RXNE bit of the Status Register becomes 1, or in other words whenever data is present in the Data Register, an interrupt is generated.

This is the first addition.

### ii.

The second addition is:

```c
NVIC_EnableIRQ(USART2_IRQn);
```

Even though USART2 can now generate an interrupt request, the Cortex-M4 processor must still be informed that it is allowed to accept interrupts from USART2.

This is done through the NVIC.

```text
NVIC
=
Nested Vector Interrupt Controller
```

The NVIC is part of the Cortex-M4 processor and is responsible for managing all interrupts.

Therefore, after enabling RXNEIE:

```text
Data arrives
↓
RXNE = 1
↓
USART2 generates interrupt request
```

But the processor will respond to this request only if the interrupt is enabled in the NVIC.

Therefore we use:

```c
NVIC_EnableIRQ(USART2_IRQn);
```

<img width="800" height="300" alt="image" src="https://github.com/user-attachments/assets/f67871a3-bc30-417d-9d4e-cf377547195f" />

Here:

- `NVIC_EnableIRQ()`

  - is a CMSIS function provided in the CMSIS headers.
  - Its purpose is to enable a particular interrupt inside the NVIC.

- The argument:

```c
USART2_IRQn
```

  - is also defined in the CMSIS device header (stm32f411xe.h).
  - It is an enumeration value that uniquely identifies the USART2 interrupt.

  Something similar to:

```c
USART2_IRQn = 38
```

(the interrupt number assigned to USART2).

Therefore:

```c
NVIC_EnableIRQ(USART2_IRQn);
```

means:

```text
Enable interrupt number corresponding to USART2
inside the NVIC
```

After this:

```text
Data arrives
↓
RXNE = 1
↓
USART2 generates interrupt request
↓
NVIC accepts request
↓
Cortex-M4 jumps to USART2_IRQHandler()
```

## USART2 Interrupt Handler

### i.

Now let's look at the Interrupt Service Routine (ISR).

This function is called whenever a USART2 interrupt occurs.

My doubt may be:

```text
We never called this function anywhere in the code. So how is it executed?
```

The answer is that this function is not called by our program. It is called automatically by the Cortex-M4 processor through the interrupt mechanism.

The flow is:

```text
Data arrives
↓
RXNE becomes 1
↓
USART2 generates interrupt request
↓
NVIC receives interrupt
↓
CPU pauses current execution
↓
CPU jumps to USART2_IRQHandler()
```

### ii.

The name:

```text
USART2_IRQHandler
```

is special.

It is already present in the Vector Table provided in the startup file.

The Vector Table contains addresses of all interrupt handlers.

Something conceptually like:

```text
USART2 Interrupt
↓
USART2_IRQHandler()
```

Therefore, when a USART2 interrupt occurs, the processor automatically looks into the Vector Table and jumps to the function named:

```text
USART2_IRQHandler()
```

That is why we never explicitly call this function in main().

### iii.

Inside the ISR:

```c
if(USART2->SR & SR_RXNE)
```

we check whether the RXNE flag is set.

RXNE means:

```text
Receive Data Register Not Empty
```

If RXNE is 1:

```text
Data has arrived
↓
Data available in USART_DR
```

Then we execute:

```c
uart_callback();
```

and transfer control to the callback function.

---

## Callback Function

### i.

This function contains the actual work that must be performed when data is received.

First:

```c
key = USART2->DR;
```

The received data is read from the Data Register and stored in the variable:

```text
key
```

Reading the Data Register also clears the RXNE flag automatically.

### ii.

Next:

```c
if(key == '1')
```

we check whether the received character is '1'.

If it is '1':

```c
GPIOA->ODR |= LED_PIN;
```

PA5 becomes HIGH.

```text
PA5 = 1
↓
LED ON
```

If any other character is received:

```c
GPIOA->ODR &= ~LED_PIN;
```

PA5 becomes LOW.

```text
PA5 = 0
↓
LED OFF
```

Since the LED is connected to PA5, the LED turns ON or OFF depending on the received character.

---

## Complete Interrupt Flow

```text
CPU doing other work
        ↓
Character arrives on PA3
        ↓
USART2 receives data
        ↓
Data stored in USART_DR
        ↓
RXNE = 1
        ↓
RXNEIE enabled
        ↓
USART2 generates interrupt
        ↓
NVIC receives interrupt
        ↓
CPU jumps to USART2_IRQHandler()
        ↓
RXNE checked
        ↓
uart_callback()
        ↓
Read USART_DR
        ↓
Store in key
        ↓
Check if key == '1'
        ↓
PA5 HIGH / LOW
        ↓
LED ON / OFF
        ↓
Return from interrupt
        ↓
CPU resumes previous execution
```
