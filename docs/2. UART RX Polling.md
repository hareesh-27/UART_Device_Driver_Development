## uart_rx_init() function

### i.

Now let's see about the uart_rx_init() function. There are only a few changes compared to UART TX Polling.

First change is that for receiving data, our USART2 module uses PA3 in Alternate Function mode.

Therefore, from the MODER register, we configure PA3 as Alternate Function mode by setting bit 7 and clearing bit 6:

```c
GPIOA->MODER &= ~(1U<<6);
GPIOA->MODER |=  (1U<<7);
```

<img width="800" height="300" alt="image" src="https://github.com/user-attachments/assets/d42443d9-151b-41a7-a8b7-8812c2a9c4e3" />

### ii.

Next, we need to configure PA3 for Alternate Function type AF07.

For PA3, the Alternate Function bits are AFRL3[3:0], which correspond to bits 15:12 in the AFRL register.

AF07 corresponds to:

```text
0111
```

Therefore, we set bits 12, 13 and 14 and clear bit 15:

```c
GPIOA->AFR[0] |=  (1U<<12);
GPIOA->AFR[0] |=  (1U<<13);
GPIOA->AFR[0] |=  (1U<<14);
GPIOA->AFR[0] &= ~(1U<<15);
```

<img width="800" height="300" alt="image" src="https://github.com/user-attachments/assets/3bdffe35-c741-4c10-9630-b7585d76ee9d" />

### iii.

After this, we enable the receiver by:

```c
USART2->CR1 = (CR1_TE | CR1_RE);
```

Here, the RE bit enables the receiver and the TE bit enables the transmitter.

These are the only changes we do in the initialization function compared to UART TX Polling. Everything else such as enabling clocks, configuring baud rate and enabling the USART2 module remains the same.

## Main Function

### i.

Now we come to the main() function.

First, we create a variable:

```c
char key;
```

This variable will be used to store the received character.

Next, we enable clock access to GPIOA:

```c
RCC->AHB1ENR |= GPIOAEN;
```

Since PA5 belongs to GPIOA, we must enable clock access to GPIOA before using PA5.

### ii.

Next, we configure PA5 as an output pin:

```c
GPIOA->MODER |=  (1U<<10);
GPIOA->MODER &= ~(1U<<11);
```

For PA5, the mode bits are:

```text
MODER5[1:0]
↓
Bit 11 and Bit 10
```

Output mode corresponds to:

```text
01
```

Therefore, we set bit 10 and clear bit 11.

We are configuring PA5 as an output because an LED is connected to PA5 on the STM32 board. We will use this LED to verify that UART reception is working correctly.

### iii.

Next, we call:

```c
uart2_rx_init();
```

As discussed previously, this function performs all the required UART configurations such as GPIO configuration, USART2 configuration, baud rate configuration, enabling the receiver and enabling the USART2 module.

After this function call, UART reception is ready.

### iv.

Next, we enter the infinite loop:

```c
while(1)
```

Inside the loop, we continuously call:

```c
key = uart2_read();
```

The purpose of this function is to wait for incoming data and read the received character from USART2.

The received character is then stored in the variable:

```c
key
```

### v.

Next, we check:

```c
if(key == '1')
```

If the received character is '1', we execute:

```c
GPIOA->ODR |= LED_PIN;
```

Here, LED_PIN corresponds to PA5.

This sets PA5 HIGH.

```text
PA5 = 1
↓
LED ON
```

Therefore, whenever the user sends the character '1' from the serial terminal, the LED connected to PA5 turns ON.

If the received character is anything other than '1', we execute:

```c
GPIOA->ODR &= ~LED_PIN;
```

This clears PA5.

```text
PA5 = 0
↓
LED OFF
```

Therefore, whenever the user sends any character other than '1', the LED connected to PA5 turns OFF.

The LED is used as a simple visual indication that UART data has been successfully received and processed.

## UART Read Function

```c
char uart2_read(void)
{
    /* Make sure receive data register is not empty */
    while(!(USART2->SR & SR_RXNE)){}

    /* Read data */
    return USART2->DR;
}
```

### i.

Now let's look at the most important function responsible for receiving data.

This function is responsible for receiving data through USART2.

Whenever data is sent from the serial terminal, it arrives at the RX pin (PA3) of USART2.

The USART hardware receives this data and stores it in the Data Register (DR).

However, before reading the Data Register, we must ensure that valid data has actually been received.

### ii.

For this purpose, we check the RXNE (Receive Data Register Not Empty) flag present in the Status Register (SR).

```c
while(!(USART2->SR & SR_RXNE)){}
```

Here:

```c
USART2->SR
```

accesses the USART Status Register.

```c
SR_RXNE
```

is a mask corresponding to bit 5 (RXNE).

According to the reference manual:

```text
RXNE = 0 → Data not received
RXNE = 1 → Received data is ready to be read
```

Therefore, the while loop continuously checks the RXNE flag.

If RXNE is 0:

```text
No data received
↓
Keep waiting
↓
Keep checking again
```

If RXNE becomes 1:

```text
Data has arrived
↓
USART hardware has placed the received data into USART_DR
↓
Data is ready to be read
↓
Exit while loop
```

This technique is called polling because the CPU continuously checks a status flag until the required event occurs.

<img width="800" height="300" alt="image" src="https://github.com/user-attachments/assets/b35d6cba-47ab-49c2-93dc-76f9549cf28d" />

### iii.

After RXNE becomes 1, the next line executes:

```c
return USART2->DR;
```

This reads the received data from the Data Register.

For example, if the user sends:

```text
1
```

from the serial terminal:

```text
Character = '1'
↓
ASCII = 49
↓
Hex = 0x31
```

The USART hardware stores:

```text
0x31
```

inside the Data Register.

When:

```c
return USART2->DR;
```

executes, the value stored in the Data Register is read and returned to the calling function.

Reading the Data Register also automatically clears the RXNE flag.

```text
Character arrives
↓
USART_DR gets data
↓
RXNE = 1
↓
CPU reads USART_DR
↓
RXNE cleared to 0
↓
Wait for next character
```

<img width="800" height="300" alt="image" src="https://github.com/user-attachments/assets/0fb013f9-e92c-4b2d-af6b-64906eba018e" />

### iv.

Back in main():

```c
key = uart2_read();
```

the returned value is stored in the variable:

```c
key
```

The program then checks whether the received character is '1' and accordingly turns the LED connected to PA5 ON or OFF.

The complete flow becomes:

```text
User sends character
        ↓
Serial Terminal
        ↓
PA3 (RX)
        ↓
USART2 Receiver
        ↓
USART_DR
        ↓
RXNE = 1
        ↓
uart2_read()
        ↓
return USART2->DR
        ↓
key variable
        ↓
if(key == '1')
        ↓
PA5 HIGH
        ↓
LED ON
```

This process repeats continuously inside the infinite loop, allowing USART2 to receive characters one by one using polling.
